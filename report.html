GitHub + Supabase · 데이터 흐름 · 업로드 방법 · 보안

전체 구조

우리 웹앱은 "두 개의 부품"으로 되어 있다

화면(프론트엔드)은 GitHub가, 데이터·로그인(백엔드)은 Supabase가 담당합니다. 이 둘을 합쳐 하나의 웹앱이 됩니다.

핵심 개념

"화면"과 "데이터 서버"는 역할이 다르다

🖥️ 프론트엔드 (GitHub Pages)
사람이 보는 화면. 버튼, 글자, 디자인, 입력칸. 그냥 파일(HTML)을 인터넷에 올려둔 것이라 서버 운영이 필요 없음.
🗄️ 백엔드 (Supabase)
눈에 안 보이는 데이터 창고 + 신분 확인소. 로그인 처리, 데이터 저장/조회를 담당. 이것도 우리가 서버를 직접 안 돌림 (BaaS).

💡 둘 다 "서버를 빌려 쓰는" 방식이라, 우리는 관리할 서버가 없습니다. 비전문가 팀에게 가장 안전한 구조입니다.

주제 1 · 왜 이 조합인가

GitHub + Supabase 를 고른 이유

  • ✓
    서버 운영 0 — 켜고 끄고 보안 패치할 서버가 없음. 누구 PC가 꺼져도 안 죽음.
  • ✓
    무료 — 소규모 사내용은 두 서비스 모두 무료 한도로 충분.
  • ✓
    흩어지지 않음 — 저장소 1개 + 데이터 1곳으로 통합. (예전처럼 사람마다 따로국밥 X)
  • ✓
    데이터를 표로 봄 — Supabase 대시보드에서 엑셀처럼 데이터 확인·수정.
  • ✓
    AI로 개발하기 쉬움 — 예제가 많아 클로드가 코드를 잘 만들어 줌.
주제 1 · 다른 방식과 비교

Netlify · Firebase 와 비교하면

구분GitHub + Supabase (우리)NetlifyFirebase
화면 호스팅무료, push로 자동배포무료, 비슷가능
데이터 저장표(SQL) — 보기 쉬움자체 DB 없음(별도 필요)JSON 트리
로그인 기능내장별도 연동 필요내장(구글 강력)
데이터 확인엑셀처럼 표로—트리라 다소 불편
적합한 경우표 형태 사내 데이터순수 화면만구글 계정 중심

핵심: Netlify는 화면 전용이라 데이터·로그인은 결국 다른 걸 붙여야 함. Firebase도 좋지만, "탭마다 표로 데이터 저장"엔 SQL인 Supabase가 직관적입니다.

주제 2 · 배포 흐름

"수정 → push" 하면 일어나는 일

내가 하는 건 ①②뿐. ③④는 자동입니다. git은 내 PC의 파일을 GitHub 창고로 안전하게 운반·기록하는 도구이고, GitHub Pages가 그걸 받아 웹사이트로 펼쳐줍니다.

주제 2 · 가장 중요한 원리 ⭐

"코드"와 "데이터"는 다른 길로 흐른다

⭐ 가장 헷갈리는 부분: 우리가 코드를 바꾸면 git→GitHub로 가지만, 사용자가 입력한 데이터(메모 등)는 git에 안 올라가고 브라우저↔Supabase로 직접 오갑니다. 그래서 데이터는 배포와 무관하게 실시간 저장됩니다.

주제 3 · 업로드 방법

로컬에서 수정하고 올리는 법

  • 1
    받기 — 「최신버전_받기.bat」 더블클릭. 다른 팀원의 최신 변경을 먼저 가져옵니다.
  • 2
    수정 — 폴더의 파일을 편집하거나, 클로드에게 “index.html에 ○○ 추가해줘”라고 부탁.
  • 3
    올리기 — 「배포.bat」 더블클릭 → 자동 commit + push → 1~2분 뒤 사이트 반영.

로컬 폴더 위치: 바탕화면 ＞ map · 사이트: https://fit-platform.github.io/map/

주제 3 · 파일은 어디로 가나

파일 종류별로 가는 곳이 다르다

무엇을어디로방법
HTML / 코드GitHub폴더에 두고 배포.bat
이미지 (로고 등)GitHubIMG/ 폴더에 넣고 배포.bat
고정 자료 (xlsx 등)GitHubdata/ 폴더에 넣고 배포.bat
사용자가 입력하는 데이터Supabase앱에서 자동 저장 (배포 불필요)

규칙: "미리 만들어 둔 파일"은 GitHub로 (배포.bat), "사람이 앱에서 입력하는 정보"는 Supabase로 (자동). ⚠️ 민감한 데이터는 공개되는 GitHub에 올리지 말고 반드시 Supabase에 저장.

주제 4 · 보안 원리

데이터는 어떻게 지켜지나

공개 키(anon key)는 그냥 "문 앞까지 오는 표"일 뿐, 데이터를 여는 열쇠가 아닙니다. 진짜 열쇠는 RLS 검문소 — 로그인한 팀원만 통과시키고, 그 외엔 전부 막습니다.

주제 4 · 우리가 적용한 보안

실제로 적용·검증한 보안 조치

  • ✓
    RLS 자물쇠 — 모든 표는 기본 "잠김". 로그인한 팀원만 읽기/쓰기 허용.
  • ✓
    로그인 필수 — 로그인 안 하면 화면조차 데이터를 못 불러옴.
  • ✓
    공개 가입 차단 — 관리자가 등록한 팀원만 로그인 가능(외부인 가입 불가).
  • ✓
    공개 키만 노출 — 코드엔 공개해도 안전한 anon 키만. 비밀키는 절대 미포함.
  • ✓
    검증 완료 — 로그아웃 상태에서 데이터 읽기 0건 + 쓰기 거부 실제 확인.

⚠️ 단, 공개 저장소(GitHub)엔 민감 데이터·비밀번호를 파일로 올리지 않기 — 데이터는 Supabase에.

✅ 요약

한 장으로 정리

• 구조: 화면(GitHub Pages) + 데이터·로그인(Supabase), 둘 다 서버 운영 0
• 장점: 무료 · 통합 관리 · 표로 데이터 확인 · AI로 개발 쉬움
• 코드 흐름: 내 PC → (배포.bat=push) → GitHub → 자동배포 → 웹
• 데이터 흐름: 브라우저 ↔ Supabase (git 안 거침, 실시간)
• 업로드: 받기.bat → 수정 → 배포.bat (1~2분 뒤 반영)
• 보안: RLS 검문소 + 로그인 + 가입 차단 (검증 완료)

이 구조 위에 탭과 기능을 계속 얹어 통합 웹앱으로 키워나갈 수 있습니다. 🚀
