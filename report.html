전체 통합 구조

여러 사람이 하나의 베이스로 앱을 만든다

핵심 개념

웹앱은 "화면"과 "데이터 서버"로 나뉜다

둘 다 "서버를 빌려 쓰는" 방식이라 우리가 직접 관리할 서버가 없습니다. 비전문가 팀에게 가장 안전한 구조입니다.

주제 1 · 왜 이 조합인가

GitHub + Supabase 를 고른 이유

  • ✓
    서버 운영 0 — 켜고 끄고 보안 패치할 서버가 없음. 누구 PC가 꺼져도 안 죽음.
  • ✓
    무료 — 소규모 사내용은 두 서비스 모두 무료 한도로 충분.
  • ✓
    흩어지지 않음 — 저장소 1개 + 데이터 1곳으로 통합. (사람마다 따로국밥 X)
  • ✓
    데이터를 표로 봄 — Supabase 대시보드에서 엑셀처럼 확인·수정.
  • ✓
    AI로 개발하기 쉬움 — 예제가 많아 클로드가 코드를 잘 만들어 줌.
주제 1 · 다른 방식과 비교

Netlify · Firebase 와 비교하면

구분GitHub + Supabase (우리)NetlifyFirebase
화면 호스팅무료 · push 자동배포무료 · 비슷가능
데이터 저장표(SQL) · 보기 쉬움자체 DB 없음JSON 트리
로그인내장별도 연동내장(구글 강력)
데이터 확인엑셀처럼 표로—트리라 불편
적합한 경우표 형태 사내 데이터순수 화면만구글 계정 중심

Netlify는 화면 전용이라 데이터·로그인은 따로 붙여야 함. Firebase도 좋지만 "탭마다 표로 데이터 저장"엔 SQL인 Supabase가 직관적입니다.

주제 2 · 배포 흐름

"수정 → push" 하면 일어나는 일

내가 하는 건 ①②뿐, ③④는 자동입니다. 1~2분 뒤 사이트에 반영됩니다.

주제 2 · 가장 중요한 원리

"코드"와 "데이터"는 다른 길로 흐른다

⭐ 가장 헷갈리는 부분: 코드는 git→GitHub로 가지만, 사용자가 입력한 데이터는 git에 안 올라가고 브라우저↔Supabase로 직접 오갑니다.

주제 3 · 업로드 방법

로컬에서 수정하고 올리는 법

  • 1
    받기 — 「최신버전_받기.bat」 더블클릭. 다른 팀원의 최신 변경을 먼저 가져옵니다.
  • 2
    수정 — 폴더의 파일을 편집하거나, 클로드에게 “index.html에 ○○ 추가해줘”라고 부탁.
  • 3
    올리기 — 「배포.bat」 더블클릭 → 자동 commit + push → 1~2분 뒤 사이트 반영.

로컬 폴더: 바탕화면 ＞ map · 사이트: https://fit-platform.github.io/map/

주제 3 · 파일은 어디로 가나

파일 종류별로 가는 곳이 다르다

무엇을어디로방법
HTML / JS / CSSGitHub폴더에 두고 배포.bat
이미지 (로고 등)GitHubIMG/ 폴더에 넣고 배포.bat
고정 자료 (xlsx 등)GitHubdata/ 폴더에 넣고 배포.bat
사용자가 입력하는 데이터Supabase앱에서 자동 저장 (배포 불필요)

규칙: "미리 만들어 둔 파일"은 GitHub로, "사람이 앱에서 입력하는 정보"는 Supabase로. ⚠️ 민감 데이터는 공개되는 GitHub에 올리지 말고 반드시 Supabase에.

주제 4 · 보안 원리

데이터는 어떻게 지켜지나

공개 키(anon key)는 "문 앞까지 오는 표"일 뿐, 데이터를 여는 열쇠가 아닙니다. 진짜 열쇠는 RLS 검문소 — 로그인한 팀원만 통과시킵니다.

주제 4 · 적용·검증한 보안

실제로 적용한 보안 조치

  • ✓
    RLS 자물쇠 — 모든 표는 기본 "잠김". 로그인한 팀원만 읽기/쓰기 허용.
  • ✓
    로그인 필수 — 로그인 안 하면 화면이 데이터를 못 불러옴.
  • ✓
    공개 가입 차단 — 관리자가 등록한 팀원만 로그인 가능.
  • ✓
    공개 키만 노출 — 코드엔 안전한 anon 키만. 비밀키는 미포함.
  • ✓
    검증 완료 — 로그아웃 상태에서 읽기 0건 + 쓰기 거부 실제 확인.
✅
요약

한 장으로 정리

• 구조: 화면(GitHub Pages) + 데이터·로그인(Supabase), 둘 다 서버 운영 0
• 협업: 여러 명이 각자 기능을 만들어 하나의 저장소에 모음
• 코드 흐름: 내 PC →(배포.bat)→ GitHub → 자동배포 → 웹
• 데이터 흐름: 브라우저 ↔ Supabase (git 안 거침, 실시간)
• 보안: RLS 검문소 + 로그인 + 가입 차단 (검증 완료)

이 구조 위에 탭과 기능을 계속 얹어 통합 웹앱으로 키워나갈 수 있습니다. 🚀

마이그레이션 · STEP 0 (PC당 처음 한 번만)

먼저, 내 PC를 GitHub에 연결한다

  • 1
    Git 설치 — git-scm.com 에서 받아 설치(기본값 Next). 내 PC가 GitHub와 대화하는 기본 도구입니다.
  • 2
    GitHub CLI 설치 — PowerShell을 열고 winget install --id GitHub.cli -e (로그인을 쉽게 해주는 도구)
  • 3
    로그인 — gh auth login → GitHub.com · HTTPS · 브라우저 로그인. ⭐ 반드시 fit-platform 계정. (한 번이면 PC에 저장됨)
  • 4
    저장소 받기(clone) — git clone https://github.com/fit-platform/map.git → 바탕화면에 map 폴더 생성(배포.bat 등 포함).

이 4단계는 PC당 딱 한 번. 어려우면 클로드에게 “이 PC를 fit-platform/map에 연결해줘” 라고 하면 설치·clone을 대신 해줍니다 (브라우저 로그인만 직접). → 끝나면 다음 페이지로.

마이그레이션 · 내 앱 옮기기

흩어진 내 앱을 현재 체계로 옮기기

  • 1
    최신 받기 — 「최신버전_받기.bat」으로 map 폴더를 최신 상태로 맞춥니다.
  • 2
    내 파일 넣기 — HTML·JS·CSS는 miniapps/, 이미지는 IMG/, 엑셀은 data/에 복사. (클로드에게 “내 앱을 miniapps/○○ 로 정리해줘”)
  • 3
    메뉴에 연결 — index.html 탭/메뉴에 내 앱 링크 추가. (클로드에게 부탁)
  • 4
    데이터는 Supabase로 ⭐ — 로컬·파일·다른 DB에 있던 데이터를 Supabase 표로 옮기고(엑셀은 CSV import), 저장 코드를 Supabase 호출로 교체. (클로드에게 부탁)
  • 5
    로그인 통합 — 따로 로그인 쓰던 앱은 공통 로그인으로 통일(이미 있는 세션 사용).
  • 6
    테스트 후 정리 — 「배포.bat」 → 로그인해 동작 확인 → 기존 Netlify·Firebase 등은 확인 후 폐기.

기억하세요: 코드·이미지·xlsx → GitHub · 사용자 데이터 → Supabase. 한 번에 다 옮기지 말고 한 앱씩 순서대로 진행하세요.
